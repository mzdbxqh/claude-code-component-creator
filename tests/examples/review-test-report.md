# 架构审阅模式测试报告

## 测试目标
验证 reviewer-core 能否正确检测反模式

## 测试用例 1: 有问题的 Skill

**文件**: `examples/sample-components/problematic-skill/SKILL.md`

### 检测结果

#### ❌ 严重问题 (1个)

**AP001: 缺少描述字段**
- 类别: yaml
- 影响: Claude Code 无法正确理解组件用途
- 修复: 添加 `description: [具体描述]`

#### ⚠️ 警告 (5个)

**AP009: 命名不规范**
- 当前: `codeReviewer` (camelCase)
- 建议: `code-reviewer` (kebab-case)

**AP003: 描述过于简短**
- 当前: "This tool reviews code." (24字符)
- 建议: 至少20字符，说明具体触发场景

**AP010: 描述过于宽泛**
- 问题: 使用了模糊词汇 "etc"
- 建议: 具体说明支持的功能

**AP005: 缺少使用示例**
- 问题: 没有 ## 示例 部分
- 建议: 添加至少1个使用示例

**AP019: 文档结构混乱**
- 问题: 缺少标准章节
- 建议: 添加工作流程、示例、常见陷阱等章节

#### ℹ️ 信息 (3个)

**AP007: 缺少反模式说明**
- 建议: 添加 ## 常见陷阱 部分

**AP013: 缺少触发条件说明**
- 建议: 在描述中明确说明触发场景

**AP015: 缺少版本声明**
- 建议: 添加 `version: 1.0.0`

### 改进建议（优先级排序）

1. **[高] 添加 description 字段**
   ```yaml
   description: 当用户请求Python、JavaScript或Java代码审查时，执行静态分析和风格检查
   ```

2. **[高] 修正组件名称**
   ```yaml
   name: code-reviewer
   ```

3. **[高] 添加使用示例**
   ```markdown
   ## 示例
   
   ### 示例1: 审查单个文件
   输入: "请审查 src/main.py"
   输出: 代码审查报告
   ```

4. **[中] 添加版本声明**
   ```yaml
   version: 1.0.0
   ```

5. **[中] 添加常见陷阱说明**
   ```markdown
   ## 常见陷阱
   1. 大型项目超时
   2. 编码问题
   ```
